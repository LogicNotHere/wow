# WoW Boosting Marketplace — бизнес-логика и флоу (MVP, версия для заказчика)

Документ описывает согласованную бизнес-логику (без архитектуры, моделей БД и API).

---

## 1) Роли и ответственность

### Customer (клиент)
- Просматривает каталог и конфигурирует услугу/лот (цены видны без логина).
- Может инициировать заказ как гость: оставляет контакт → Admin связывается (вне системы) → после оплаты Admin создаёт заказ в системе и отправляет гостю **magic link** для просмотра статуса/деталей. Чат по magic link — post‑MVP.
- Может создавать заказ из аккаунта (если зарегистрирован) — тогда заказ виден в кабинете.
- Общается в чате заказа (для авторизованных), получает инструкции по оплате и выполнению.
- Следит за статусами/таймлайном, оставляет публичный отзыв после закрытия.
- Может запросить отмену/спор по правилам возвратов.

### Booster (исполнитель)
- Любой пользователь может заполнить анкету/профиль бустера (Booster profile).
- Анкета бустера на MVP: базовые поля (например имя + ссылка на Discord). Admin получает нотификацию и апрувит выдачу роли/допуска “Booster”.
- Видит **только оплаченные** заказы в пуле (факт оплаты может быть скрыт: в UI можно показывать статус как `Created`).
- Берёт заказ:
  - обычный бустер: запрос “взять заказ” → `NeedsAdminReview` → апрув Admin → `Accepted`
  - `SuperBooster`: `Paid → Accepted` без апрува Admin
- Ведёт прогресс и чекпоинты (опционально с фото), выбирает персонажа для выполнения.

### Operator / Content Manager (контент/слоты) — опциональная роль
Нужна, если хотим делегировать монотонную работу без выдачи полного Admin.
- Создание/редактирование лотов и заполнение контента.
- Без доступа к подтверждению оплат, возвратам, банам пользователей, финальным решениям по спорам.

### Admin (оператор системы)
- Управляет каталогом/ценами/коэффициентами, промо/скидками и витринными тегами.
- Координирует заказ: коммуникация, подтверждение оплаты, назначение/апрув бустеров, контроль статусов.
- Обрабатывает споры/возвраты (manual), ведёт внутренние заметки по Customer, банит пользователей.

---

## 2) Каталог услуг

Категории (MVP): `Raid`, `Mythic+`, `PvP`, `Leveling`, `Coaching`, `Gold`, `Collections`, `Achiv`, `Professions`.

Каждый лот/услуга:
- находится в дереве категорий (категории могут быть вложенными); лоты прикрепляются только к конечным узлам (листьям)
- имеет опции (сложность/кол-во ранов/параметры прогресса и т.д.)
- имеет режим выполнения:
  - MVP: **Self-play** или **Piloted** (опция на лоте)
  - post‑MVP: удалённое выполнение/assist через **AnyDesk** (по логике как piloted)
- может иметь скидку и витринные метки `HIT/HOT` (теги без влияния на цену)

---

## 3) Ценообразование (backend)

### Принципы
- Цена считается **только на backend**; клиент цену не передаёт.
- Валюта MVP: **EUR**.
- При создании заказа фиксируется **snapshot цены** (чтобы изменения прайса в каталоге не меняли уже созданный заказ).
- Если после `Paid` нужно изменить состав/лот/опции — это делает Admin вручную; доплата/пересчёт тоже решаются вручную (MVP).

### Компоненты цены (обобщённо)
- **Base price** лота.
- **Primary driver** (главный параметр): например M+ уровень/кол-во ранов, Raid сложность/боссы, PvP рейтинг, Leveling уровни, Coaching часы.
- **Модификаторы/опции**: доп.опции конкретного лота.
- **Piloted multiplier**: множитель для piloted, настраиваемый Admin по категории и/или лоту.

### Скидки/промо/теги
- Admin может задавать скидки на категорию и/или конкретный лот; ручная скидка ограничена до **-50%** и требует причины.
- Скидки **не суммируются**: если лот уже “со скидкой”, то промокоды/прочие скидки не применяются.
- `HIT/HOT` — витринные теги, **не влияют** на цену.

---

## 4) Оплата (manual)

- Для зарегистрированного Customer заказ создаётся **до оплаты**, но выполнение **не начинается** до подтверждения оплаты.
- Для гостя до оплаты существует только “заявка/контакт”; сам заказ Admin создаёт **после оплаты**.
- `PaymentPending → Paid` подтверждается **вручную Admin**.
- TTL ожидания оплаты (как быстро авто-отменять неоплаченные) — на MVP не фиксирован (регламент/ручные действия Admin).

## 5) End-to-end флоу заказа

### Customer flow
1) Просмотр каталога → выбор лота → выбор опций → расчёт цены (backend).
2) **Вариант A (из аккаунта):** логин/регистрация → создание заказа (`Created`) → открывается чат `customer↔admin`.
3) Admin согласует детали и выдаёт инструкции оплаты → `PaymentPending`.
4) После подтверждения оплаты Admin переводит заказ в `Paid`.
5) **Вариант B (гостевой):** пользователь оставляет контакт и нажимает “оплатить” → Admin связывается **вне системы** → после оплаты Admin создаёт заказ в системе (как минимум сразу в `Paid`) и отправляет гостю **magic link**.
6) Назначение исполнителя:
   - ручное назначение Admin: `Paid → Assigned → (booster принимает) Accepted`
   - инициатива бустером: `Paid → NeedsAdminReview → (Admin approve) Accepted`
   - `SuperBooster`: `Paid → Accepted` без апрува
7) При `Accepted` booster добавляется в общий чат заказа (для авторизованного Customer; для гостевого заказа коммуникация с Customer остаётся вне системы).
8) Выполнение: `InProgress` → `Done` → Admin закрывает `Closed`.
9) После `Closed` клиент оставляет публичный отзыв/оценку.

### Booster flow (после `Paid`)
- Видит пул оплаченных заказов:
  - по умолчанию — “профильные” (по навыкам/категориям)
  - отдельный список “все заказы (обзор)” остаётся в MVP
- Берёт заказ:
  - обычный бустер: `Paid → NeedsAdminReview` (требует апрув Admin)
  - `SuperBooster`: `Paid → Accepted` сразу
- Перед стартом (опционально) выбирает персонажа:
  - на MVP — ручной ввод (интеграции raider.io не планируется; см. out of scope)

## 6) Статусы заказа (state machine)

**Основные:**
- `Created` — заказ создан (обычно из аккаунта) и ожидает согласования/оплаты.
- `PaymentPending` — Admin согласует детали и ожидает оплату/подтверждение оплаты (manual).
- `Paid` — оплата подтверждена Admin; заказ попадает в пул бустеров (в UI бустера можно маскировать как `Created`).
- `NeedsAdminReview` — обычный бустер запросил “взять заказ”; Admin должен approve/decline (decline → обратно `Paid`).
- `Assigned` — Admin вручную назначил конкретного бустера.
- `Accepted` — бустер принял заказ (после `Assigned`, после апрува `NeedsAdminReview`, или сразу `SuperBooster`).
- `RequirementsPending` — сбор обязательных вводных перед стартом (для piloted — доступы/инструкции; для бустера — персонаж и т.д.).
- `InProgress` — выполнение начато.
- `Done` — исполнитель отметил выполнение завершённым (и/или Admin подтвердил).
- `Closed` — Admin закрыл заказ; после этого доступен отзыв.

Примечание: в гостевом сценарии заказ может появляться в системе сразу со статусом `Paid` (так как создаётся Admin после подтверждения оплаты).

**Проблемные ветки:**
- `Canceled` — заказ отменён.
- `Disputed` — открыт спор, решение принимает Admin.
- `Refunded` / `PartialRefund` — оформлен полный/частичный возврат.

---

## 7) Коммуникация и уведомления

### Чат
- Один чат на заказ (для авторизованных): сначала `customer↔admin`, booster добавляется при `Accepted`.
- Private чат `admin↔booster` внутри системы — backlog (на старте возможно общение через внешние каналы).

### Уведомления (направления)
- Системные уведомления (по событиям):
  - Admin:
    - in-app (в админке) + realtime через socket (MVP можно как заглушку)
- Маркетинговые рассылки — email (opt-in/unsubscribe — TBD).

Минимум для MVP:
- in-app уведомления Admin + realtime обновление через socket (MVP можно как заглушку).

## 8) Отмены, споры, возвраты (MVP)

- До `Paid`: отмена без последствий.
- После `Paid` и до `Accepted`: **100% возврат**.
- После `Accepted` и до `InProgress`: отмена/возврат — manual решение Admin.
- На `InProgress`: возврат/компенсация — manual решение Admin с опорой на прогресс/чекпоинты.
- После `Done/Closed`: только через `Disputed` и решение Admin.

`Piloted`:
- На MVP правила те же, что и для обычных заказов.
- Доступы передаются после `Paid`, первым получает Admin; доступ видит только назначенный исполнитель.
- Post‑MVP: для `Piloted` требуется стрим (всегда) — исполнитель кидает ссылку в чат; опционально embed Twitch в карточке заказа.

## 9) MVP: что входит / что не входит

### In scope (MVP)
- Каталог + опции + backend прайсинг (EUR) + snapshot цены.
- Ручное подтверждение оплаты Admin.
- Пул оплаченных заказов для бустеров + `NeedsAdminReview` + ручное `Assigned` + `SuperBooster` авто-accept.
- Один чат заказа (booster добавляется при `Accepted`).
- Таймлайн событий, чекпоинты (опционально фото).
- Публичные отзывы после `Closed`.
- Скидки на лот/категорию (не суммируются) + `HIT/HOT` теги.
- Возвраты/споры manual по правилам выше.
- Гостевой заказ + **magic link** для просмотра статуса/деталей.

### Out of scope (позже / optional)
- Автоподтверждение оплаты провайдером.
- Мультивалютность.
- Private чат `admin↔booster` внутри системы.
- Автоматические правила частичных возвратов.
- Customer-слоты времени/переносы времени для Customer.
- Автоматическое повышение/ачивки для `SuperBooster` (если решим вообще вводить).
- Доп. режим выполнения через удалённый доступ (например **AnyDesk**) как отдельная опция лота (по логике как piloted: данные через Admin → исполнитель).
- `Piloted`-стриминг: отправка ссылки на стрим в чат (и опциональный embed окна Twitch внутри заказа).
- Календарь для части лотов (в первую очередь Raid): расписание от `RaidLeader` + выбор слота Customer после оплаты (очень поздняя фича).
- Уведомления бустерам по рейдам из календаря: за 30 минут до начала (каналы: Discord/Telegram/email — TBD).
- Чат по magic link (авторизация токеном, почти как обычный чат; post‑MVP).
- Витринные страницы лотов: `ServicePage` как набор контентных блоков (`PageBlock`) с типом и `payload` (JSON) + валидация по схеме типа + публикация `published` (page builder в админке).
- Интеграция raider.io (скорее всего нет).
- Интеграция отзывов через Trustpilot/аналоги (если решим — отдельный проект).

### Выплаты бустерам
- MVP: выплаты вне системы (вручную).
- После MVP: возможна пайаут-логика (процент/фикс) и отображение заработка (валюта — TBD).

## 10) Решения после созвона (последние правки)

1) **Гостевой заказ:** выдаём гостю **magic link** для просмотра статуса/деталей без регистрации. Чат по magic link — post‑MVP.
2) **Пайауты/заработок в кабинете бустера:** бустер **ничего не видит**; выплаты полностью на стороне Admin (договорённости вне системы).
3) **Нотификации Admin (MVP):** in-app (в админке) + realtime через socket (MVP можно как заглушку); интеграции Discord/Telegram — post‑MVP.
4) **Нотификации бустерам по календарю (post‑MVP):** за **30** минут до начала рейда (каналы: Discord/Telegram/email — TBD).
